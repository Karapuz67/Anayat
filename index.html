💡 Почему трёхслойная?
  • ✅ Разделение ответственности — каждый слой делает своё дело
  • ✅ Масштабируемость — легко добавить новые функции
  • ✅ Тестируемость — каждый слой можно тестировать отдельно
  • ✅ Гибкость — можно заменить любой слой (например, поменять БД)

⭐ CLTV — Customer Lifetime Value
Что это? Сколько денег один клиент принесёт пекарне за всё время, пока он остаётся клиентом

Формула

CLTV = средний заказ × частота × время жизни
     = AvgOrderValue × Frequency × Lifetime

Пример

Жан Дюпон:
  Средний заказ:  100 ₸
  Частота:        5 раз/месяц
  Активен:        30 месяцев
  ─────────────────────────
  CLTV = 100 × 5 × 30 = 15,000 ₸

Почему важно?

  • ✅ Показывает ценность каждого клиента
  • ✅ Помогает решить — VIP или обычный клиент
  • ✅ Показывает, сколько вложить в маркетинг
  • ✅ Помогает приоритизировать клиентов

📝 Часть 1: Общее понимание

Q1: Что такое проект Belle Croissant Lyonnais? ▼
Belle Croissant Lyonnais — это полнофункциональная система управления пекарней, разработанная на .NET 8. Она включает: управление товарами (круассаны, хлеб, тарты), управление клиентами, систему учёта заказов и бизнес-аналитику.

Система состоит из трёх частей: REST API (сервер), Desktop приложение Windows Forms (клиент) и система тестирования (19 unit-тестов).
Q2: Какая архитектура и почему? ▼
Используется трёхслойная архитектура:

Desktop (Presentation) → REST API (Application) → SQL Server (Data)

Почему трёхслойная: ✅ Разделение ответственности ✅ Масштабируемость — легко добавить новые функции ✅ Тестируемость — каждый слой тестируется отдельно ✅ Гибкость — можно заменить любой слой

Q7: Все API endpoints для Products ▼
GET    /api/products              → все товары
GET    /api/products/1            → товар с ID=1
GET    /api/products?category=Bread → фильтр
POST   /api/products              → создать товар
PUT    /api/products/1            → обновить товар
DELETE /api/products/1            → удалить товар
Q8: Валидации при создании Product ▼
При создании товара проверяется:

❌ ProductName — не может быть пустым
❌ Category — только "Pastry", "Bread" или "Tart"
❌ Price — должна быть больше 0
❌ Cost — должна быть > 0 и МЕНЬШЕ Price
❌ ReleaseDate — обязательна

Если нарушить — вернётся ошибка 400 Bad Request
Q9: Что означает HTTP код 201? ▼
201 Created — означает, что новый объект успешно создан.

Возвращается при POST запросе (например, после создания нового товара). В ответе обычно содержится JSON с данными созданного объекта, включая его ID.
Q10: Все HTTP коды ответов ▼
КодСтатусКогда возникает
200 OK✅GET запрос успешен
201 Created✅POST создал объект
400 Bad Request❌Неправильные данные (нарушена валидация)
401 Unauthorized❌Нет или неверный логин/пароль
404 Not Found❌Объект с таким ID не найден
500 Server Error❌Ошибка на сервере
Q11: Как работает аутентификация? ▼
Используется Basic Auth:

Логин: staff, Пароль: BCLyon2024

Как работает внутри:
staff:BCLyon2024
    ↓ Base64 кодирование
c3RhZmY6QkNMeW9uMjAyNA==
    ↓ добавляется в заголовок HTTP
Authorization: Basic c3RhZmY6QkNMeW9uMjAyNA==
API проверяет каждый запрос — есть ли правильный заголовок.

📊 Часть 3: Модели данных

Q14: Поля модели Product ▼
ProductId, ProductName*, Category*, Price*, Cost*, Seasonal, Active, ReleaseDate*, Ingredients (опц.), Description (опц.)

Категории: только Pastry, Bread, Tart
* — обязательные поля
Q15: Поля модели Customer ▼
CustomerId, FirstName, LastName, Email, PhoneNumber, Age, Gender, ZipCode, MembershipStatus (Basic/Gold/Platinum), JoinDate, LastPurchaseDate, TotalSpending, AvgOrderValue, Frequency, PreferredCategory, ChurnStatus
Q16: Поля модели Order и статусы ▼
OrderId, CustomerId, OrderDate, TotalAmount, Status, DeliveryMethod (Pickup/Delivery), OrderItems

Статусы: Pending → Processing → Completed (или Cancelled)
Q17: Что такое OrderItem? ▼
OrderItem — позиция в заказе. Поля: ItemId, OrderId (ссылка на заказ), ProductId (ссылка на товар), Quantity (количество), UnitPrice (цена за единицу).

Пример: 3× Круассан по 2.50 = сумма 7.50
Q18: Связи между моделями ▼
Customer (1) ──→ (много) Order
Order     (1) ──→ (много) OrderItem
OrderItem (много) ──→ (1) Product

Один клиент → много заказов
Один заказ → много позиций  
Каждая позиция → один товар
Q19: Что такое Entity Framework? ▼
Entity Framework (EF) — это ORM (Object-Relational Mapper).

Простыми словами: EF позволяет работать с базой данных через C# объекты, без написания SQL вручную.

// Вместо: SELECT * FROM Products WHERE Category = 'Pastry'
// Пишем на C#:
var pastries = _db.Products.Where(p => p.Category == "Pastry").ToList();
Q20: Что такое DbContext? ▼
DbContext — главный класс Entity Framework, который управляет подключением к базе данных и всеми таблицами.

class AppDbContext : DbContext {
    public DbSet<Product>   Products   { get; set; }
    public DbSet<Customer>  Customers  { get; set; }
    public DbSet<Order>     Orders     { get; set; }
    public DbSet<OrderItem> OrderItems { get; set; }
}

Q22: Что такое ApiService? ▼
ApiService — класс в Desktop приложении, который отвечает за отправку HTTP запросов в API.

Когда сотрудник нажимает кнопку "Добавить товар", форма вызывает ApiService, который формирует HTTP POST запрос и отправляет его на localhost:5001.
Q23: Что такое async/await? ▼
async/await — асинхронное программирование.

Без async/await: приложение "зависает" пока ждёт ответа от API (пользователь видит зависший экран).

С async/await: приложение остаётся отзывчивым — пока идёт запрос, можно делать другие вещи.

// Асинхронный запрос — не блокирует UI
var products = await _apiService.GetProductsAsync();
Q24: Процесс добавления товара через Desktop ▼
1. Сотрудник заполняет форму и нажимает "Добавить"
2. ProductsForm вызывает ApiService.CreateProduct(dto)
3. ApiService отправляет POST на http://localhost:5001/api/products
4. ProductsController проверяет данные (валидация)
5. Entity Framework сохраняет в SQL Server
6. Возвращается 201 Created с данными товара
7. Desktop показывает "Товар добавлен!" и обновляет таблицу
Q25: Статусы заказа и их смысл ▼
Pending — заказ только создан, ждёт обработки
Processing — заказ в работе, пекарня готовит
Completed — заказ выполнен и выдан
Cancelled — заказ отменён

Переход: Pending → Processing → Completed (или Cancelled)

📈 Часть 5: Analytics

Q26: Что такое CLTV и зачем? ▼
CLTV = Customer Lifetime Value — сколько денег клиент принесёт за всё время.

Формула: CLTV = AvgOrderValue × Frequency × CustomerLifetime

Пример: Жан Дюпон: 100₸ × 5раз/мес × 30мес = 15,000₸

Важность: показывает ценность клиента, помогает решить — VIP или нет, сколько тратить на удержание.
Q27: Что такое Churn и как интерпретировать? ▼
Churn Rate — % клиентов которые ушли и перестали покупать.

Формула: Churn Rate = ушедшие / всего × 100%

Интерпретация: 10–20% — норма ✅, 30% — нужно улучшать ⚠️, 50%+ — критично ❌

Что делать при высоком Churn: программы лояльности, улучшить качество, специальные предложения.
Q28: Как Analytics интегрирована из Session 1? ▼
Python (Session 1) → CSV файлы → C# AnalyticsDataLoader → C# Models → AnalyticsForm

CSV файлы: Session1_CLTV.csv, Session1_Churn_Analysis.csv, Session1_SalesForecast.csv

AnalyticsDataLoader читает CSV и создаёт объекты CLTVData, ChurnMetrics, SalesForecast, которые отображаются в 3 вкладках AnalyticsForm.

Тест: себестоимость не может быть выше цены
[Test]
public void Constructor_CostGreaterThanPrice_ThrowsArgumentException()
{
    // Price=1.00, Cost=2.00 → ошибка! Пекарня в убытке!
    Assert.Throws<ArgumentException>(() =>
        new BakeryItem(1, "Круассан", "Pastry", 1.00m, 2.00m, 10));
    // Тест проходит если выброшено исключение ✅
}
